## SQL 写入可靠性结论

本轮最关键的 SQL 相关验证有两点：

1. MySQL 重复主键 Upsert 语义测试通过：提交 2 条，实际 2 条，完整性 100%。
2. PostgreSQL 重复主键 Upsert 语义测试通过：提交 2 条，实际 2 条，完整性 100%。

这说明当前实现已经能覆盖前面讨论的核心问题：

- PostgreSQL 不再固定只能用 schema 第一列作为冲突键。
- PostgreSQL 同批次重复 key 不再导致 upsert 报错。
- `ConflictUpdate` 不更新冲突列。
- `ConflictReplace` 在 PostgreSQL 上使用 upsert 覆盖语义。
- MySQL `ConflictUpdate` 和 `ConflictReplace` 仍保持数据库原生语义。

## 需要注意的边界

这次是 30 秒级轻量压测，适合验证功能正确性、端到端流程和基础性能形态，不等同于生产容量上限。

不能直接从本报告推导：

- 单台生产数据库极限吞吐。
- 长时间运行数小时后的稳定性。
- 网络抖动、磁盘 IO 饱和、数据库锁竞争下的表现。
- 多 schema、大 row、复杂索引、真实业务事务下的性能。
- 不同云数据库规格下的吞吐上限。

## 下一轮建议

建议增加三类压测，作为发布前或性能基线使用：

1. 长稳压测：`TEST_DURATION=30m` 或 `1h`，观察错误率、GC、内存和吞吐漂移。
2. 参数矩阵：分别测试 `BATCH_SIZE=100/200/500/1000`，找出 MySQL/PostgreSQL 的吞吐和延迟平衡点。
3. 业务模型压测：使用更接近真实业务的 row 大小、唯一键组合、索引数量和 update columns。

推荐命令：

```bash
TEST_DURATION=30m CONCURRENT_WORKERS=20 RECORDS_PER_WORKER=20000 BATCH_SIZE=500 BUFFER_SIZE=10000 ./scripts/run_stress_report.sh
```

如果只想验证 PostgreSQL/MySQL update/replace，可先跑：

```bash
TEST_DURATION=5m CONCURRENT_WORKERS=10 RECORDS_PER_WORKER=5000 ./scripts/run_stress_report.sh --backend postgres --backend mysql
```
